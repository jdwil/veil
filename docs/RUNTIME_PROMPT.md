# veil-runtime Implementation Prompt

## Context

You are implementing `veil-runtime` — the thin bootstrap layer that runs VEIL-generated Rust code. The project lives at `~/dev/jd/veil-runtime`.

VEIL generates complete domain code (aggregates, ports/traits, adapters, services) but does NOT generate:
1. A concrete `Bus` implementation (the inter-context message router)
2. A `main()` entry point that wires everything together

Your job is to provide both.

## What VEIL Generates

When you run `veil gen some_app.veil -o output/`, you get a Rust workspace with:
- **Domain types** (structs, enums, value objects with invariants)
- **Ports** (async traits like `CustomerRepo`, `Bus`)
- **Adapters** (impl blocks for the traits, e.g. `PgCustomerRepo`)
- **Application services** (async functions that take `&Deps` and execute business logic)
- **A `Deps` struct** per module: `pub struct Deps { pub bus: Arc<dyn Bus + Send + Sync>, pub customer_repo: Arc<dyn CustomerRepo + Send + Sync>, ... }`
- **A `veil_shared` crate** with `Bus` trait, `DomainError`, and shared infrastructure

The `Bus` trait (generated in `veil_shared`):
```rust
#[async_trait]
pub trait Bus: Send + Sync {
    async fn dispatch(&self, evt: String) -> Result<(), DomainError>;
    async fn invoke(&self, cmd: String) -> Result<String, DomainError>;
    async fn request(&self, qry: String) -> Result<String, DomainError>;
}
```

Messages are serialized as formatted strings (v1). The Bus receives strings like:
- `"GetCohort { id: \"abc-123\" }"` (from `request GetCohort{id: cohort_id}`)
- `"CreateCohort { id: \"def-456\", tenant_id: \"xyz\", name: \"Test\" }"` (from `invoke CreateCohort{...}`)
- `"CustomerCreated { id: \"abc\", email: \"x@y.com\" }"` (from `dispatch CustomerCreated{...}`)

## What You Need to Implement

### 1. `InProcessBus`

A Bus implementation that routes messages to registered handlers within the same process. This is the default for monolithic deployments and testing.

```rust
pub struct InProcessBus {
    handlers: HashMap<String, Box<dyn Handler>>,
}

#[async_trait]
trait Handler: Send + Sync {
    async fn handle(&self, payload: String) -> Result<String, DomainError>;
}
```

The bus needs to:
- Parse the message name from the string (e.g. "GetCohort" from `"GetCohort { id: ... }"`)
- Look up the registered handler for that message
- Call the handler with the payload
- Return the result

For `dispatch` (fire-and-forget events): log + call handlers, don't wait for response.
For `invoke` (commands): call handler, return result.
For `request` (queries): call handler, return result.

### 2. Handler Registration

The generated application services (e.g. `HandleGetCohort`, `HandleAddCohortMember`) need to be registered with the Bus. Provide a builder pattern:

```rust
let bus = InProcessBus::builder()
    .register("GetCohort", GetCohortHandler::new(deps.clone()))
    .register("CreateCohort", CreateCohortHandler::new(deps.clone()))
    .build();
```

### 3. `main()` Harness / Entry Point

A configurable entry point that:
1. Reads `DATABASE_URL` from environment
2. Creates a `PgPool` (via sqlx)
3. Constructs adapter instances (PgTenantRepo, PgCohortRepo, etc.)
4. Creates the InProcessBus with handlers registered
5. Builds the `Deps` struct with all dependencies
6. Starts an HTTP server (axum) exposing the Bus over REST

### 4. HTTP Interface (Optional but recommended)

Expose the Bus over HTTP so other services can send commands/queries:

```
POST /bus/invoke   { "message": "CreateCohort", "payload": "{...}" }
POST /bus/request  { "message": "GetCohort", "payload": "{...}" }
POST /bus/dispatch { "message": "CustomerCreated", "payload": "{...}" }
```

## Architecture Principles

- **VEIL generates domain code, veil-runtime runs it.** The runtime never contains domain logic.
- **The Bus is the only inter-context communication mechanism.** Contexts never import each other's types directly.
- **CQRS is enforced:** Commands accept an ID (client-generated UUID), return only success/failure. Queries return typed DTOs.
- **The Bus payload format is strings (v1).** Future versions may use `serde_json::Value` or typed enums. For now, parse the message name from the first word before `{`.
- **Adapters are generated by VEIL.** The runtime just constructs them with their dependencies (pool, config).
